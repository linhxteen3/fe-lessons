FE試験対策・Yasashii Ch.5 (全章) + Slide 5_システム構成要素 第1-3回

📘 Phần 1 — Tóm tắt lý thuyết

1.1. システムとは・処理形態による分類 (Lesson 05-01)

🇯🇵 日本語 システムとは、これまで手作業で行っていた仕事を、コンピュータを使って自動化するための仕組みである。システムの処理形態は大きく 集中処理 と 分散処理 の2種類に分類される。

集中処理：1台の大型コンピュータ（ホストコンピュータ）に複数の端末を接続し、端末から送られてくるすべての命令をその1台が処理する形態。

分散処理：ネットワークでつながれた複数のコンピュータが、処理を分担して行うシステム。FE試験では分散処理に関する知識が問われる。
🇻🇳 Tiếng Việt Hệ thống (システム) là cơ chế tự động hoá những công việc trước đây làm thủ công bằng máy tính. Hệ thống được phân thành 2 loại: xử lý tập trung (集中処理) và xử lý phân tán (分散処理).

集中処理: 1 máy tính lớn (ホストコンピュータ) xử lý mọi lệnh từ các terminal kết nối tới.

分散処理: nhiều máy tính nối qua mạng cùng chia sẻ xử lý. FE thường hỏi loại này.

1.2. 分散処理システムの3種類 (Lesson 05-01)

🇯🇵 日本語 (1) クライアントサーバシステム：サービスを利用する側を「クライアント」、サービスを提供する側を「サーバ」と呼ぶ。

(2) 3層クライアントサーバシステム：プレゼンテーション層（画面表示）、アプリケーション層（業務処理）、データベース層（データ管理）の3層に分離した構成。

(3) Webシステム：クライアント側のソフトウェアにWebブラウザを使い、HTTPで通信する。業務処理はサーバ側で実行し、クライアントはHTMLの記述に従って画面に表示する。
🇻🇳 Tiếng Việt (1) Client-Server: client = bên dùng dịch vụ, server = bên cung cấp.

(2) 3-tier Client-Server: chia 3 lớp — lớp giao diện (presentation), lớp ứng dụng (application/業務処理), lớp dữ liệu (database).

(3) Webシステム: client dùng browser, giao tiếp HTTP. Xử lý nghiệp vụ ở server, client chỉ hiển thị theo HTML.

1.3. ⭐ システムの二重化 (Slide 第1回 + Yasashii 過去問 5-2)

🇯🇵 日本語 業務上必要不可欠なシステムを停止させないため、複数のシステムを用意して信頼性を高める設計を 二重化（冗長化）と呼ぶ。

デュアルシステム：同じ処理を2組のシステムで同時に独立に行い、結果を照合する方式。1組が誤動作しても照合により検出可能。ただし処理能力は2倍にならない。

デュプレックスシステム：同じ処理に対して2組のシステムを用意し、1組は処理中（現用系）、もう1組は待機系（予備）として待機する方式。現用系に障害が発生したら待機系へ切り替える。
🇻🇳 Tiếng Việt Twin-hoá (二重化): chuẩn bị nhiều hệ thống để duy trì hoạt động ngay cả khi 1 cái hỏng — gọi là "redundancy" (冗長化).

Dual System (デュアル): 2 hệ thống chạy cùng lúc độc lập, so sánh kết quả. Phát hiện lỗi qua so sánh. Lưu ý: năng lực xử lý KHÔNG tăng gấp đôi.

Duplex System (デュプレックス): 1 hệ thống làm việc (現用系), 1 hệ thống chờ (待機系). Khi 現用 hỏng → chuyển sang 待機系.
🇬🇧 Dual = 2 systems run in parallel and cross-check; Duplex = active + standby (failover).

1.4. スタンバイ方式の種類 (Yasashii 過去問 5-3)

🇯🇵 日本語 ホットスタンバイシステム：待機系のコンピュータに、現用系と同一のオンライン処理プログラムをあらかじめ起動して待機させておく方式。現用系が故障したとき、速やかに待機系に切り替えて処理を続行できる。

コールドスタンバイシステム：待機系には電源を入れず、停止した状態で待機させておく方式。故障時に待機系を起動してから切り替えるため、切替に時間がかかる。
🇻🇳 Tiếng Việt Hot Standby: máy 待機系 luôn bật điện, program đã được khởi động sẵn. Khi 現用系 hỏng → chuyển sang nhanh chóng (速やかに).

Cold Standby: máy 待機系 tắt điện, chỉ bật khi cần. Chuyển đổi tốn nhiều thời gian hơn.

1.5. シンクライアント (Slide 第1回)

🇯🇵 日本語 シンクライアントシステム（Thin Client）：クライアント端末の機能を必要最低限にとどめ、サーバ側でアプリの実行やデータの管理を行う方式。HDDやSSDを省き、Excel等もインストールしない。

最大のメリット：PC端末を紛失しても情報が漏洩しない（データはサーバ側にあるため）。

実現方法：VDI（自社サーバに仮想デスクトップを作成）、DaaS（外部事業者のクラウドに仮想デスクトップ作成）。
🇻🇳 Tiếng Việt Thin Client: máy client chỉ có chức năng tối thiểu, mọi xử lý + data đều ở server. Không có HDD/SSD, không cài Excel...

Ưu điểm lớn nhất: mất máy không lộ data (data ở server).

2 cách triển khai: VDI (virtual desktop trên server công ty), DaaS (virtual desktop trên cloud bên ngoài).

1.6. データのバックアップ方式 (Slide 第1回)

🇯🇵 日本語 フルバックアップ：対象となるデータを全てバックアップする方式。

差分バックアップ：フルバックアップ以降に変更されたデータをバックアップする方式。

増分バックアップ：直前のバックアップ以降に変更されたデータをバックアップする方式。
🇻🇳 Tiếng Việt Full backup: backup toàn bộ data mỗi lần.

Differential (差分): backup data thay đổi kể từ Full backup gần nhất.

Incremental (増分): backup data thay đổi kể từ backup trước đó (bất kỳ loại).
📝 Phân biệt nhanh / 区別のコツ: 差分 luôn so với full; 増分 luôn so với backup trước đó. Vì vậy 差分 tăng dần, 増分 cố định nhỏ.

1.7. ⭐ RASIS — システムの評価指標 (Slide 第2回)

🇯🇵 日本語 RASISとは、システムを評価する5つの指標の頭文字。
  • Reliability（信頼性）：システムが安定して稼働し続けられること → MTBFで評価
  • Availability（可用性）：使いたい時にいつでも使えること → 稼働率で評価
  • Serviceability（保守性）：故障した時に早く復旧できること → MTTRで評価
  • Integrity（完全性）：データの正確性
  • Security（安全性）：セキュリティ
🇻🇳 Tiếng Việt RASIS là 5 chỉ số đánh giá hệ thống:
  • Reliability (độ tin cậy): hệ thống chạy ổn định liên tục → đo bằng MTBF
  • Availability (khả dụng): dùng được bất cứ lúc nào → đo bằng 稼働率
  • Serviceability (bảo trì): sửa lỗi nhanh → đo bằng MTTR
  • Integrity (chính xác): tính đúng của data
  • Security (an toàn): bảo mật

1.8. ⭐⭐ MTBF と MTTR (Slide 第2回 — トレンタム)

🇯🇵 日本語 MTBF（Mean Time Between Failures）：システムの故障と故障の間隔の平均時間（＝システムが正常に稼働している時間の平均）。値が大きいほど信頼性が高い。

MTTR（Mean Time To Repair）：システムが故障した時に修理にかかる平均時間。値が小さいほど修理が早く保守性が高い。
🇻🇳 Tiếng Việt MTBF = thời gian trung bình giữa các lần hỏng (= thời gian hoạt động bình thường trung bình). MTBF càng lớn → tin cậy càng cao.

MTTR = thời gian trung bình để sửa chữa khi hỏng. MTTR càng nhỏ → bảo trì càng tốt.
MTBF = (システムが稼働していた時間の合計) ÷ (故障した回数)
MTTR = (故障中の修理にかかった時間の合計) ÷ (故障した回数)

1.9. ⭐ 稼働率（Availability）— Slide 第2回

🇯🇵 日本語 稼働率とは、システムの全運転時間のうち、システムが正常に稼働している時間の割合。MTBFとMTTRを使って次の式で求める：
🇻🇳 Tiếng Việt 稼働率 (Availability) là tỷ lệ thời gian hệ thống chạy bình thường so với tổng thời gian. Tính bằng MTBF và MTTR theo công thức:
稼働率 = MTBF / (MTBF + MTTR)
= 正常稼働時間 / (正常稼働時間 + 修理時間)
時間軸の例: [ 稼働 200h ][ 修理 3h ][ 稼働 100h ][ 修理 1h ][ 稼働 150h ][ 修理 2h ] ←──── 稼働時間の平均 = MTBF ────→ ←─── 修理時間の平均 = MTTR ───→
🇯🇵 上の例：MTBF = (200+100+150)/3 = 150時間
MTTR = (3+1+2)/3 = 2時間
稼働率 = 150/(150+2) = 150/152 ≈ 0.987（98.7%）
🇻🇳 Theo ví dụ trên: MTBF = (200+100+150)/3 = 150 giờ
MTTR = (3+1+2)/3 = 2 giờ
稼働率 = 150/152 ≈ 0.987 (98.7%)

1.10. ⭐ 複数システムの稼働率 (Slide 第2回)

🇯🇵 日本語 複数のシステムから構成される場合、直列システムか並列システムかによって全体の稼働率が異なる。

直列システム：システム全体が稼働するのは、システムAとシステムBの両方が稼働している時のみ。

並列システム：システム全体が稼働するのは、システムAとシステムBの少なくとも一方が稼働している時。
🇻🇳 Tiếng Việt Khi hệ thống được tạo từ nhiều hệ thống con, công thức tính稼働率 toàn bộ phụ thuộc vào cách ghép:

Series (直列): toàn hệ thống chỉ hoạt động khi CẢ 2 (A AND B) đều ổn.

Parallel (並列): toàn hệ thống hoạt động khi ÍT NHẤT 1 (A OR B) ổn. Hỏng khi cả 2 cùng hỏng.
直列システムの稼働率 = A × B
並列システムの稼働率 = 1 − (1 − A)(1 − B)
📝 直感的な理解 / Cách hiểu trực giác: 直列 = AND → 掛け算 / nhân; 並列 = OR → 「両方とも壊れる確率」を1から引く / 1 trừ "cả hai cùng hỏng".

1.11. 信頼性設計の用語 (Yasashii Lesson 05-02)

用語🇯🇵 説明🇻🇳 Giải thích
フォールトトレラント
(Fault Tolerant)		 システムが部分的に故障しても、システム全体としては必要な機能を維持する設計。冗長化で対応。 Hỏng 1 phần nhưng toàn hệ thống vẫn duy trì chức năng cần thiết (dựa vào冗長化/redundancy).
フェールソフト
(Fail Soft)		 障害が起きた時に、性能が落ちてもシステム全体は動作させる設計。限定動作を「縮退運転（フォールバック）」と呼ぶ。 Hỏng thì giảm hiệu năng vẫn chạy. Trạng thái vận hành hạn chế = 縮退運転 / fallback.
フェールセーフ
(Fail Safe)		 障害が起きた時に、システムを止めてでも安全を優先させる設計。 Hỏng thì dừng cũng được, ưu tiên an toàn (vd robot công nghiệp dừng khi có người).
フールプルーフ
(Fool Proof)		 人間が間違った使い方をしても、システムに異常が起こらないようにする設計。 Người dùng thao tác sai vẫn không gây lỗi hệ thống (vd lò vi sóng không nóng khi cửa hở).

1.12. ストライピング・ミラーリング (Yasashii Lesson 05-02)

🇯🇵 日本語 ストライピング（RAID 0）：データを分散して複数の磁気ディスクに書き込むことによって、データ入出力の高速化を図る方式。ディスクが1つでも壊れると動作しなくなる。

ミラーリング（RAID 1）：複数のハードディスクに同じデータを書き込むことによって、システムの信頼性を高める方式。同じディスク容量で比較した場合、保存できるデータ量はストライピングの半分。
🇻🇳 Tiếng Việt Striping (RAID 0): phân tán data trên nhiều ổ đĩa → tăng tốc độ I/O. Nhưng 1 ổ hỏng = toàn bộ hỏng.

Mirroring (RAID 1): ghi cùng data vào nhiều ổ → tăng độ tin cậy. Dung lượng hiệu dụng chỉ = 50% so với striping cùng tổng dung lượng.

1.13. システムの性能指標 (Yasashii Lesson 05-03)

指標🇯🇵 説明🇻🇳 Giải thích
スループット コンピュータが単位時間当たりに処理する仕事の量。 Khối lượng công việc xử lý trong 1 đơn vị thời gian.
ターンアラウンドタイム 「入力 + 処理 + 出力」の全工程にかかる合計時間。 Tổng thời gian "input + xử lý + output" — toàn bộ quy trình.
レスポンスタイム
（応答時間）		 処理だけにかかる時間。 Chỉ thời gian xử lý (không tính input/output).
ベンチマークテスト 使用目的に合わせて選定した標準的なプログラムやデータを実行させ、システムの処理性能を測定するテスト。 Đo hiệu năng bằng cách chạy chương trình/data chuẩn phù hợp với mục đích sử dụng.

1.14. キャパシティプランニング (Yasashii Lesson 05-03)

🇯🇵 日本語 キャパシティプランニングとは、システムの稼働状況を継続的に評価して、システムに求められる処理能力を計画することである。処理能力を向上する必要がある場合、主に次の2手段を取る：

スケールアウト：サーバの台数を増やすことで、システム全体の処理能力を向上させる。

スケールアップ：個々のサーバの処理能力を向上させる（CPUを高性能なものに換装、主記憶装置を増設等）。
🇻🇳 Tiếng Việt Capacity Planning: liên tục đánh giá tình trạng vận hành để lên kế hoạch năng lực xử lý cần thiết. Khi cần tăng năng lực:

Scale Out: tăng số lượng server để mở rộng năng lực tổng.

Scale Up: tăng năng lực từng server (đổi CPU mạnh hơn, thêm RAM...).

例題 2.1: MTBF・MTTR・稼働率の計算 (Slide第2回 原文)

問題 / Đề bài
🇯🇵 あるシステムの稼働履歴は以下のとおりである：稼働 200h → 修理 3h → 稼働 100h → 修理 1h → 稼働 150h → 修理 2h。MTBF・MTTR・稼働率を求めよ。
🇻🇳 Lịch sử vận hành: 稼働 200h → 修理 3h → 稼働 100h → 修理 1h → 稼働 150h → 修理 2h. Tính MTBF, MTTR, 稼働率.
MTBF = (200+100+150) ÷ 3 = 150 時間
MTTR = (3+1+2) ÷ 3 = 2 時間
稼働率 = 150 ÷ (150+2) = 150/152 ≈ 0.987 (98.7%)

例題 2.2: 直列システムの稼働率 (Slide第2回 原文)

問題 / Đề bài
🇯🇵 稼働率0.9のシステムA と 稼働率0.9のシステムB を直列接続したときの全体の稼働率を求めよ。
🇻🇳 Nối tuần tự A (0.9) và B (0.9). Tính稼働率 tổng.
直列の稼働率 = A × B = 0.9 × 0.9 = 0.81 (81%)
🇯🇵 注：直列の稼働率は必ず各システム単体より低くなる（両方が稼働している必要があるため）。
🇻🇳 Lưu ý: 稼働率 直列 luôn thấp hơn từng thành phần (vì cần CẢ 2 cùng ổn).

例題 2.3: 並列システムの稼働率 (Slide第2回 原文)

問題 / Đề bài
🇯🇵 同じくA=0.9、B=0.9を並列接続したときの全体の稼働率を求めよ。
🇻🇳 Cùng A=0.9, B=0.9 nối song song. Tính稼働率 tổng.
並列の稼働率 = 1 − (1−0.9)(1−0.9) = 1 − 0.01 = 0.99 (99%)
🇯🇵 注：並列の稼働率は必ず各システム単体より高くなる（少なくとも一方が稼働すれば全体が稼働するため）。これが冗長化で信頼性を高める理由。
🇻🇳 Lưu ý: 稼働率 並列 luôn cao hơn từng thành phần (chỉ cần ÍT NHẤT 1 ổn). Đây là lý do dùng redundancy.

例題 2.4: MTBFを逆算する問題

問題 / Đề bài
🇯🇵 あるシステムの稼働率は0.95、MTTRは5時間である。MTBFを求めよ。
🇻🇳 稼働率 = 0.95, MTTR = 5h. Tính MTBF.
0.95 = MTBF / (MTBF + 5)
0.95 (MTBF + 5) = MTBF → 0.95 MTBF + 4.75 = MTBF
0.05 MTBF = 4.75 → MTBF = 95 時間

例題 2.5: 直列+並列の組合せ

問題 / Đề bài
🇯🇵 システムA（稼働率0.9）を直列で、システムB（0.8）とシステムC（0.8）の並列群と接続する。全体の稼働率を求めよ。
🇻🇳 A (0.9) nối tuần tự với nhóm B-C song song (B=0.8, C=0.8). Tính tổng.
手順1: 並列BC = 1 − (1−0.8)² = 1 − 0.04 = 0.96
手順2: A × BC = 0.9 × 0.96 = 0.864

例題 2.6: Yasashii 過去問 5-2 (デュアル vs デュプレックス)

過去問 / Đề thi cũ (平成31年度)
🇯🇵 冗長構成におけるデュアルシステムの説明として、適切なものはどれか。
ア．2系統のシステムで並列処理をすることによって性能を上げる方式である。
イ．2系統のシステムの負荷が均等になるように、処理を分散する方式である。
ウ．現用系と待機系の2系統のシステムで構成され、現用系に障害が生じたときに、待機系が処理を受け継ぐ方式である。
エ．一つの処理を2系統のシステムで独立に行い、結果を照合する方式である。
🇻🇳 Đáp án đúng: エ. デュアルシステム = 「2 hệ thống xử lý độc lập cùng 1 việc rồi so kết quả」. Lưu ý: ウ là デュプレックス (現用 + 待機, failover), không phải デュアル.
